## 2026-07-11 · M3（格式互转：Office/HTML ↔ PDF）

### ✅ 完成项
- **Word 转 PDF**（`word-to-pdf.ts`）：mammoth 将 .docx 转为带基础排版的 HTML → `htmlToPdf` 栅格化为 PDF；支持纵向/横向。
- **Excel 转 PDF**（`excel-to-pdf.ts`）：SheetJS 读取工作簿，每个工作表转 HTML 表格 → 栅格化为一份 PDF（默认横向更适合宽表）。
- **PDF 转 Word（文本级）**（`pdf-to-word.ts`）：pdf.js 按页提取文本层（新增 `extractPageText` 按行聚合）→ docx 生成 .docx，每页插入分页；适合可复制文本型 PDF。
- **HTML 转 PDF**（`html-to-pdf.ts`）：读取 .html 文本 → 离屏宿主 → 栅格化 PDF。
- 新增 `core/html2pdf.ts`：`htmlToPdf(el, opts)`（DOM→多页 PDF，A4 纵向/横向、边距）+ `createRenderHost(html, orientation)`（离屏渲染宿主，尺寸按 A4 比例 794/1123px）。
- 4 个工具已注册到 `src/tools/index.ts`，UI 自动出现入口（含方向选择字段）。

### ⚠️ 报错 / 踩坑
- 首次构建主包 2.5MB（html2canvas+jspdf+xlsx+docx+mammoth 全被急切打包）。
- 修复：① `vite.config.ts` 加 `manualChunks` 拆分 pdfjs/pdflib/html2pdf/office 独立 chunk；② 4 个 M3 工具改为在 `run()` 内动态 `import()` 重型库，使其变为**按需异步 chunk**。最终主包仅 22.8KB，office(1.24MB)/html2pdf(560KB) 等仅在对应工具首次运行时拉取。

- 沙箱无法启动 Chrome，M3 运行时（mammoth 转换、html2canvas 栅格化、docx 生成）未做真机验证，需本地 `npm run dev` 实际跑一遍。

### 🔧 可优化点 / 待办
- Word/Excel→PDF 为栅格化，文本不可选、图片/单元格样式可能丢失 → M5 可考虑服务端高保真（如 LibreOffice headless / 商业 API）。

- M3 工具依赖 DOM 未入 Worker，大文件处理时主线程可能卡顿（可接受 MVP）。

### 📌 决策
- M3 范围确定为「纯前端、栅格化/文本级」MVP，不引入服务端，符合 Cloudflare Pages 零后端定位。

## 2026-07-12 · 部署踩坑：`wrangler deploy` 误用

> 用户按「方法二（Dashboard）」部署，构建成功但部署步骤报 `Missing entry-point to Worker script or to assets directory`，并提示 "run `wrangler pages deploy` instead"。

### ⚠️ 原因
- 构建本身 OK（`npm run build` → `dist/` 生成）。失败在部署步：执行的是 **Workers** 命令 `npx wrangler deploy`，而 `wrangler.toml` 声明的是 **Pages** 项目（`pages_build_output_dir = "dist"`）。Pages 必须用 `wrangler pages deploy dist`，不能用 `wrangler deploy`。
- 推断用户在 Dashboard 的**构建命令**里写了 `npx wrangler deploy`（或 `npm run build && npx wrangler deploy`）。Dashboard 方式本应只填 `npm run build`，它会自动部署 `dist/`，无需手动 wrangler。
- 报错建议加 `main = "src/index.ts"` / `compatibility_date` 是 **Worker** 配置，本项目纯静态，**不能照做**。

### ✅ 修复（已写进 README_部署.md）
- 方法二：Dashboard 构建命令**只填 `npm run build`**、输出目录 `dist`、环境变量 `NODE_VERSION=22`；删掉任何 wrangler 命令后重新部署。
- 方法一（CLI）：用 `npx wrangler pages deploy dist`（不是 `wrangler deploy`）。
- 排错表新增该条目 + 醒目警告框。

### 📌 决策
- 这是用户侧配置理解偏差，非代码问题；不改源码，仅完善手册与日志。

#### 🔁 同日补刀（看清日志后修正）

- 若走 Dashboard「方式二」则更简单：构建命令只填 `npm run build`，Pages 自动部署，完全不用 wrangler。
- 已在手册顶部加「⚠️ 一句话避坑」醒目框，把两种正确写法钉死。

#### 🔧 再补刀（用户贴出 Dashboard 实际界面后纠正）
- **关键发现**：用户用的是 **新版 Pages Build configuration** 界面，字段为 `Build command`(可选) / `Deploy command`(**必填**) / `Non-production branch deploy command`(可选)，**没有独立的「构建输出目录」字段**。
- 因此之前"Deploy command 留空、Pages 自动部署"的建议是**基于旧界面，对新界面不适用**。新界面 Deploy command 必填，正确值就是 `npx wrangler pages deploy dist`（带 `pages` + `dist` 参数）。

#### 🔓 再再补刀（Deploy command 正确后遇到认证错）
- 用户改用 `npx wrangler pages deploy dist` 后，构建通过、命令正确执行，但报 `Authentication error [code: 10000]`，日志显示 `authenticating Wrangler via ... CLOUDFLARE_API_TOKEN`。
- **根因**：构建环境变量里设了 `CLOUDFLARE_API_TOKEN`，它**覆盖**了 Pages 构建环境自带的自动凭据；而该 token 缺少「Cloudflare Pages: Edit」权限 → 调用 `/accounts/.../pages/projects/pdf-toolkit` 失败。
- **账号角色是 "Super Administrator - All Privileges" 不等于 token 有该 scope**——API token 权限是单独授予的。
- **修法**：删除构建环境变量里的 `CLOUDFLARE_API_TOKEN`（保留 `NODE_VERSION=22`），重新部署；Pages 会用自带凭据自动授权。或换一个带 `Cloudflare Pages: Edit` 权限的新 token。
- 已把该坑写进 README「方式二」环境变量说明 + 排错表。部署命令本身（`npx wrangler pages deploy dist`）已验证正确。

#### 🔎 用户贴出真实界面后大纠正（前两轮认证分析部分失真）
- 用户实际界面是**新版「Worker 式」构建设置**：标题 "Connect your **Worker** to a Git repository"，字段含 `Build command` / `Deploy command` / `Version command: npx wrangler versions upload` / **`API token`（显示 `pdf-toolkit build token`）** / `Variables and secrets: None`。
- **关键纠正**：`Variables and secrets` 是 **None**，根本没有 `CLOUDFLARE_API_TOKEN` 环境变量！我前一轮"删除 CLOUDFLARE_API_TOKEN"的建议是**错的**（用户也反馈没找到）。
- **真正的认证载体是构建设置里的 `API token` 字段（即 `pdf-toolkit build token`）**。它缺少「Cloudflare Pages: Edit」权限 → 调 pages API 报 `code: 10000`。

## 2026-07-12（续）· 二次修复：批量崩溃 + 加密彻底无效

> 用户重测后反馈：多文件批量仍报 `出错： errors is not defined`（压缩/转图/提取文本/Word/Excel/HTML），且**单文件正常**；PDF 加密"未实际加密"。

### 🔍 根因（与上一轮不同，是更底层的两处）
1. **`errors is not defined` 崩溃**：`ui.ts` 的 `errors` 数组声明在批量分支（`else` 块）内部，却在函数外层 `if (allResults.length === 0)` 处引用 → 块级作用域 ReferenceError。只要批量工具抛错、allResults 为空就必崩。
2. **多文件批量工具真实失败（崩溃的隐藏诱因）**：`pdfjs` 文档（缩略图渲染 + 工具内 `loadPdf`）用完从不 `destroy()`，多文件累积耗尽 pdf.js worker → 后续 `loadPdf` 抛错、allResults 为空 → 触发上面的崩溃。压缩/转图/提取文本/PDF→Word 全部中招。
3. **加密无效（本轮彻底解决）**：原版 `pdf-lib` 的 `save({userPassword,ownerPassword})` **静默忽略密码**（源码根本不引用 userPassword，不产生 `/Encrypt`）。上一轮"owner 留空默认=user"的改法无效，因为 pdf-lib 压根没实现加密。Node 已验证：保存后无 `/Encrypt`，无密码也能打开。

### ✅ 本轮修复
- `ui.ts`：把 `const errors = []` 提升到 `runTool` 的 try 作用域顶部；批量与单文件失败都能正确显示首个失败原因。缩略图渲染后 `doc.destroy()` 释放 pdf.js 文档。
- `encrypt.ts`：改用 **`@cantoo/pdf-lib`**（pdf-lib 分支，含标准安全处理器）。流程 `load → doc.encrypt({userPassword, ownerPassword, permissions}) → save()`。Node 验证：生成 `/Encrypt`、无密码打不开、用户/owner 密码可开、错密被拒、可正常解密。设 `workerSafe:false`。
- `compress / pdf-to-image / pdf-to-text / pdf-to-word`：处理完 `doc.destroy().catch(()=>{})`，避免多文件 worker 泄漏。

### 📦 部署
- `npm run build` 通过，@cantoo/pdf-lib 进入 pdflib chunk。
- 沙箱无 GitHub SSH key / Cloudflare token，**需用户在本地终端**：`wrangler pages deploy dist`（或 `git push` 触发自动部署）。

### ⚠️ 仍待真机确认
- Excel/HTML→PDF 用 `html2canvas`（不走 pdfjs），本轮主要修的是 pdfjs 泄漏 + 崩溃；若它们多文件仍失败，新代码会显示**真实错误信息**，重测时直接看状态栏即可定位。

## 2026-07-12 · 三轮修复：批量 .call 崩溃 + 解密无效 + 多密码方案

> 用户重测反馈：压缩/转图/提取文本/Word/Excel/HTML **多文件** 报 `X.call is not a function`（单文件正常）；PDF 解密 单/多文件 均报"密码错误"（密码其实正确）。

### 🔍 根因
1. **批量 `.call is not a function`**：`ui.ts` 批量循环（`else` 分支）调用 `executeTool(t, [file], options, { onProgress: ... })` —— 第 4 参传成了**对象**，而 `executeTool` 期望的是**函数** `(r,label)=>void`。工具内 `ctx.onProgress(...)` 实际调用的是这个对象 → `X.call is not a function`（每个工具 minified 变量名不同，于是看到 a/o/d/x/u/h.call 各异）。单文件分支传的是函数，所以正常。
2. **解密永远"密码错误"**：`decrypt.ts` 用的是原版 `pdf-lib`，而原版**不实现加解密**，任何加密 PDF 加载都失败被统一判为"密码错误"。此外即使解开，`save()` 也会把 /Encrypt 原样写回，输出仍带密码。
3. **多文件不同密码**：原设计全局只有一个 `password` 字段，批量时所有文件共用，无法区分。

### ✅ 本轮修复
- `ui.ts`：批量循环第 4 参改为真正的箭头函数 `(r,label)=>setProgress(...)`。
- `decrypt.ts`：改用 `@cantoo/pdf-lib`。流程 `load(bytes,{password})` → 解锁后 `copyPages` 复制到全新 `PDFDocument` → `save()`，输出**真正无密码**（Node 已验证：/Encrypt 消失、无密码可打开、错密被拒）。
- **多密码支持（顺带解决用户的"多文件不同密码"疑问）**：`password` 字段现在接受多个密码，用逗号 / 空格 / 换行分隔；`run()` 对每个文件依次尝试所有密码，命中即解。这样多文件各带不同密码时，用户把所有密码都填进去即可，无需逐次操作。

### 📦 部署
- `npm run build` 通过；@cantoo 进入 pdflib chunk，decrypt 逻辑进入 worker chunk。
- 沙箱无 SSH key / Cloudflare token，需用户在本地：`wrangler pages deploy dist`（或 `git push` 触发自动部署）。

### ⚠️ 待真机确认
- 压缩/转图/提取文本/Word/Excel/HTML 多文件：本轮修复 onProgress 传参，应全部恢复。
- PDF 解密：单文件、多文件（含不同密码）均应正常。
- 建议回归：加密→解密闭环（用本工具加密的 PDF，再用本工具解密应能打开且无密码）。
