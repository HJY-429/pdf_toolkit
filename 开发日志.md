## 2026-07-11 · M1 + M2 落地

### ✅ 完成项
- 升级核心架构：
  - `core/types.ts`：`Tool` 接口改为统一 `ToolInput{ files, options }` / `ToolOutput[]`；新增 `fields`（UI 自动渲染表单）、`multiple`。
  - 新增 `core/io.ts`（下载/多文件下载/命名）、`core/pdfjs.ts`（PDF.js 封装，worker 经 Vite `?url` 引入）。
- M1：重写 `merge`（支持多文件+排序）、`rotate`；新增 `split`（每页/固定页数/自定义范围）。
- M2：新增 `compress`（pdf.js 重绘降清压缩）、`encrypt`/`decrypt`（pdf-lib 密码）、`watermark`/`pagenumber`（pdf-lib 绘字）、`image-to-pdf`（pdf-lib 嵌图）、`pdf-to-image`/`pdf-to-text`（pdf.js）。
- 通用 UI `ui/ui.ts`：选择工具 → 上传（点击/拖拽）→ 参数表单 → 进度条 → 结果下载（单文件直接下载链接，多文件支持"全部下载"）；多文件显示 ↑↓ 排序与删除。
- 注册中心 `tools/index.ts` 汇总 11 个工具。
- `npm install && npm run build` **通过**：218 模块，pdf.worker 独立打包为 1.38MB mjs，主包 810KB（gzip 292KB）。

### ⚠️ 报错 / 踩坑
- 浏览器实测（agent-browser/Chrome）在本沙箱被 macOS Mach-port 限制阻断，无法实时渲染验证 UI；改用 `vite build` 通过 + 代码审查替代。真机 `npm run dev` 仍需人工验证交互。
- `pdfjs-dist` v4 的 worker 必须用 Vite `?url` 引入并赋值 `GlobalWorkerOptions.workerSrc`，否则运行期报 "Setting up fake worker"。已在 `core/pdfjs.ts` 处理。
- 构建提示 chunk >500KB 警告：因 pdf-lib + pdf.js 体积大，**仅为警告非错误**；M4 可做按需 import 拆分。

### 🔧 可优化点 / 待办
- M4（下一阶段）：拖拽排序已做，待加 pdf.js 缩略图预览、批量并行、Web Worker 分块、单文件大小限制与提示、空结果友好提示。
- 中文水印/页码：需嵌入 CJK 字体（权衡体积，后续按需）。
- 压缩对纯矢量 PDF 可能变大：考虑先判页面是否含图片再决定策略，或提供"仅重采样图片"模式。
- 大文件 OOM 风险：M4 引入 Worker + 进度分块。
- 部署：暂未接 Cloudflare；`vite build` 产物在 `dist/`，后续按文档 7.x 用 `wrangler` 部署 Pages。

### 📌 决策
- 记录方式采用**双文档**：`PDF处理网站开发文档.md`（设计总纲，仅更新状态/决策）+ `开发日志.md`（流水细节）。比单文档混写更清晰。
- 基本交互模型：上传 → 处理 → 下载，符合用户预期；保留 ToolRegistry 以便 M3/M5 持续扩展。

## 2026-07-11 · M4（UI 打磨 / 产品化）

### ✅ 完成项
- **缩略图预览**：文件列表中 PDF 用 pdf.js 渲染首页（0.25 缩放）、图片用原生对象 URL；按 name+size 缓存去重。
- **单文件大小限制**：`MAX_FILE_MB=50`，超限文件被跳过并提示「已跳过 N 个超过 50MB 的文件」。
- **空结果友好提示**：结果区为空时显示说明文案；批处理部分失败时状态栏标注「N 个文件处理失败已跳过」。
- **批量处理**：单文件工具（加密/水印/PDF→图片 等）现在支持一次选多个文件，逐个处理并汇总下载；多文件工具（合并/拆分）仍整体处理并保留 ↑↓ 排序。
- **PWA 离线**：新增 `public/manifest.webmanifest`、`public/icon.svg`、`public/sw.js`（运行时缓存：导航网络优先+回退首页，静态资源 cache-first）；`main.ts` 仅在 `import.meta.env.PROD` 注册 SW；`index.html` 链接 manifest。
- **Web Worker 分块**：纯 pdf-lib 工具（merge/split/rotate/encrypt/decrypt/watermark/pagenumber/image-to-pdf）标记 `workerSafe`，经 `src/worker/runner.ts` 在 Worker 中执行，主线程保持响应；Worker 异常（含 60s 超时）自动回退主线程。`Tool` 接口新增 `workerSafe?` 字段。

### ⚠️ 报错 / 踩坑
- 构建通过（218 模块）；Worker 正确拆分为独立 chunk `runner-*.js`（436KB），pdf.worker 仍 1.38MB。
- 沙箱仍无法启动 Chrome，UI 交互与 Worker 运行时行为未做真机验证；`vite build` 通过 + 代码审查替代，需本地 `npm run dev` 实际验证。
- 已知风险：单次 Worker 调用异常会回退主线程，不影响可用性；若 Worker 运行时崩溃（非工具报错）会从主线程兜底，性能略降。

### 🔧 可优化点 / 待办
- 缩略图对每个选中文件都渲染；大批量时可做懒加载 / 列表虚拟化。
- Worker 目前仅覆盖 pdf-lib 类工具；pdf.js 类（压缩/转图片/提取文本）仍在主线程，后续可研究 pdf.js 在 Worker 内嵌子 Worker 方案。
- 50MB 上限为硬编码常量，后续可做成设置项或按工具差异化。
- PWA 未做「新版本提示刷新」，SW 更新需用户手动刷新（MVP 可接受）。

### 📌 决策
- M4 范围：拖拽排序（M1 已做）+ 缩略图 + 批量 + 大小限制 + 空结果提示 + PWA + Worker 分块，全部落地。
- Worker 采用「工具自声明 workerSafe + UI 按声明路由 + 异常回退主线程」策略，最大化兼容性与安全性。

## 2026-07-11 · M3（格式互转：Office/HTML ↔ PDF）

### ✅ 完成项
- **Word 转 PDF**（`word-to-pdf.ts`）：mammoth 将 .docx 转为带基础排版的 HTML → `htmlToPdf` 栅格化为 PDF；支持纵向/横向。
- **Excel 转 PDF**（`excel-to-pdf.ts`）：SheetJS 读取工作簿，每个工作表转 HTML 表格 → 栅格化为一份 PDF（默认横向更适合宽表）。
- **PDF 转 Word（文本级）**（`pdf-to-word.ts`）：pdf.js 按页提取文本层（新增 `extractPageText` 按行聚合）→ docx 生成 .docx，每页插入分页；适合可复制文本型 PDF。
- **HTML 转 PDF**（`html-to-pdf.ts`）：读取 .html 文本 → 离屏宿主 → 栅格化 PDF。
- 新增 `core/html2pdf.ts`：`htmlToPdf(el, opts)`（DOM→多页 PDF，A4 纵向/横向、边距）+ `createRenderHost(html, orientation)`（离屏渲染宿主，尺寸按 A4 比例 794/1123px）。
- 4 个工具已注册到 `src/tools/index.ts`，UI 自动出现入口（含方向选择字段）。

### ⚠️ 报错 / 踩坑
- 首次构建主包 2.5MB（html2canvas+jspdf+xlsx+docx+mammoth 全被急切打包）。
- 修复：① `vite.config.ts` 加 `manualChunks` 拆分 pdfjs/pdflib/html2pdf/office 独立 chunk；② 4 个 M3 工具改为在 `run()` 内动态 `import()` 重型库，使其变为**按需异步 chunk**。最终主包仅 22.8KB，office(1.24MB)/html2pdf(560KB) 等仅在对应工具首次运行时拉取。
- 仍有 chunk >500KB 构建警告（vendor 体积所致），为 advisory，不影响功能；Cloudflare Pages 静态托管无碍。
- 沙箱无法启动 Chrome，M3 运行时（mammoth 转换、html2canvas 栅格化、docx 生成）未做真机验证，需本地 `npm run dev` 实际跑一遍。

### 🔧 可优化点 / 待办
- Word/Excel→PDF 为栅格化，文本不可选、图片/单元格样式可能丢失 → M5 可考虑服务端高保真（如 LibreOffice headless / 商业 API）。
- PDF→Word 为文本级，无排版/图片/字体 → 如需高保真需复杂布局重建或服务端方案。
- HTML→PDF 无法加载外部 CSS/图片（html2canvas 限制）→ 可提示用户使用自包含 HTML，或改用打印路径。
- M3 工具依赖 DOM 未入 Worker，大文件处理时主线程可能卡顿（可接受 MVP）。

### 📌 决策
- M3 范围确定为「纯前端、栅格化/文本级」MVP，不引入服务端，符合 Cloudflare Pages 零后端定位。
- 重型库统一采用「动态 import + manualChunks」双策略，兼顾首屏速度与按需加载。
